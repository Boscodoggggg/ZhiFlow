# 智流 Lite 爆品设计文档

## 1. 一句话定位

智流 Lite 是一个本地优先的多 Agent 编码交付台，帮助开发者从需求生成 Spec，拆分子任务，并行调度多个 AI 编码 Agent，在独立 Git worktree 中实现、交叉评审，并最终由人类确认后创建 PR。

英文开源定位建议：

```text
ZhiFlow Lite: Local-first command center for parallel AI coding agents.
Spec, split, run, review, and PR.
```

中文传播定位建议：

```text
从 Spec 到 PR 的多 Agent 编码流水线。
```

核心记忆点：

```text
Human plans. Agents implement in parallel. Agents review each other. Human ships.
```

产品语言原则：

```text
默认中文界面，内建国际化能力，开源传播中英双语。
```

第一版 UI 默认使用简体中文，因为它更符合当前设计、开发和首批中文用户验证节奏；但代码、文案、日期时间、状态枚举和 Agent 模板必须从第一版开始支持国际化，避免后续重构。

## 2. 为什么它有机会成为爆品

当前 AI 编码工具已经能写代码，但开发者真正卡住的是流程控制：

- 一个复杂需求如何拆成多个可并行任务。
- 多个 Agent 如何避免互相污染仓库。
- Agent 写完后谁来审查、如何确认它没有偏离需求。
- 多个 Diff、测试结果、风险点如何汇总成人能判断的交付结果。
- 最后如何安全地 Commit、Push、创建 PR。

智流 Lite 不做通用项目管理平台，也不做新的 AI 编码模型。它只做一件事：把已有 Codex、Claude Code、Gemini CLI、Cursor Agent、OpenCode 等编码工具编排成可控的本地交付流水线。

1k+ star 的关键不是功能多，而是第一眼能让开发者明白：

```text
我可以把一个需求拆给多个 AI 同时做，而且每个 Agent 都在隔离 worktree 里，写完还有别的 Agent 帮我 Review。
```

## 3. 第一批目标用户

优先服务这三类用户：

- 已经每天使用 Codex、Claude Code、Gemini CLI、Cursor Agent 的独立开发者。
- 小团队 Tech Lead，需要同时推进多个小任务，但不放心 AI 直接改主工作区。
- 开源项目维护者，希望把 issue、需求、bugfix 变成可审查的 AI 实现 PR。

暂不优先服务：

- 需要复杂权限、组织架构、报表和审批流的大企业。
- 不使用 Git 或不使用命令行 AI 编码工具的普通项目成员。
- 只想要聊天式 AI 助手的轻量用户。

## 4. 产品边界

### 4.1 必须做

- 本地启动：`npx zhiflow` 或 `zhiflow board`。
- Repo-native 状态：Spec、Plan、Tasks、Run、Review 记录保存在仓库 `.zhiflow/` 中。
- Git worktree 隔离：每个子任务默认一个独立 worktree。
- Agent 编排：自动检测 Codex、Claude Code、Gemini CLI 等本机 CLI。
- 并行执行：多个子任务可同时分配给不同 Agent。
- 多 Agent 评审：实现 Agent 完成后，可触发一个或多个只读 Reviewer Agent。
- 人类门禁：Commit、Push、PR、合并前必须由人确认。
- WebUI 工作台：一屏展示 Spec、Kanban、Agent Run、Diff、Review。
- 中文默认界面与国际化架构：首版默认 `zh-CN`，同时内置 `en-US` 文案包和语言切换入口。

### 4.2 第一版不做

- 不做 SaaS。
- 不做团队成员和复杂权限。
- 不做完整测试管理、发布管理、知识库管理。
- 不做企业级审计系统。
- 不自研 AI 编码 Agent。
- 不自动合并到主分支。
- 不让 Reviewer Agent 默认直接修改代码。

## 5. 爆品 Demo 路径

第一支演示视频必须 60 秒内讲完：

```text
1. 在一个已有 Git 仓库运行 npx zhiflow
2. 输入需求：新增邮箱密码登录
3. ZhiFlow 生成 spec.md、plan.md、tasks.md
4. 系统拆成 3 个子任务：API、UI、测试
5. 三个 Agent 分别进入三个 worktree 并行实现
6. 右侧实时显示每个 Agent 的日志、文件变更、测试结果
7. 实现完成后，Codex 写的让 Claude Review，Claude 写的让 Gemini Review
8. ZhiFlow 汇总：2 个通过，1 个要求修改
9. 人点击“让实现 Agent 修复”
10. 修复完成后人点击“创建 PR”
```

这个 Demo 的传播标题建议：

```text
I let 3 AI coding agents implement one feature in parallel, then review each other before opening a PR.
```

## 6. 核心工作流

### 6.1 从需求到任务

```text
User intent
-> Explore
-> Spec
-> Plan
-> Tasks
-> Task graph
```

每个任务必须有：

- 标题。
- 背景。
- 输入上下文。
- 验收标准。
- 预计修改范围。
- 依赖关系。
- 推荐 Agent。
- 风险等级。

### 6.2 多 Agent 并行实现

```text
Task group
-> create worktree per task
-> assign implementer agent
-> run agents concurrently
-> stream logs
-> capture diff and tests
-> mark ready for review
```

并行执行原则：

- 每个子任务只能修改自己的 worktree。
- 同一文件被多个任务修改时标记潜在冲突。
- 默认并发数不超过 3，避免机器资源和模型额度失控。
- 用户可以暂停、取消、重试任意 Agent Run。

### 6.3 多 Agent 交叉评审

```text
Implementation complete
-> collect diff, tests, logs, spec
-> run reviewer agents in read-only mode
-> generate review reports
-> aggregate decisions
-> human gate
```

Reviewer Agent 默认只读，不能直接改代码。它只接收：

- Spec。
- Plan。
- Task。
- 验收标准。
- Diff。
- 变更文件摘要。
- 测试输出。
- 实现 Agent 总结。

Reviewer 输出：

```text
decision: approve | request_changes | block
risk_level: low | medium | high
findings:
  - file
  - line
  - severity
  - reason
  - suggested_fix
summary
```

### 6.4 单工具模式与多工具模式

多 Agent 不等于必须安装多个不同厂商的 Agent。智流 Lite 的设计必须把“Agent 角色”和“Agent Provider”解耦：

```text
Agent Role: Implementer / Spec Reviewer / Code Reviewer / Test Reviewer / Security Reviewer
Agent Provider: Codex / Claude Code / Gemini CLI / OpenCode / Cursor Agent
```

也就是说，用户只有 Codex 也能跑：

```text
Codex 实现 Agent
-> Codex 规格评审 Agent
-> Codex 代码评审 Agent
-> 人工门禁
```

用户只有 Claude Code 也能跑：

```text
Claude 实现 Agent
-> Claude 测试评审 Agent
-> Claude 安全评审 Agent
-> 人工门禁
```

推荐能力分级：

| 模式 | 依赖 | 产品表现 | 信任强度 |
| --- | --- | --- | --- |
| 单 Agent 基础模式 | 只有一个 CLI | 一个实现会话 + 一个只读复核会话 | 基础 |
| 单 Provider 多角色模式 | 只有 Codex 或只有 Claude | 同一 Provider 运行多个不同角色和 Prompt | 中 |
| 多 Provider 交叉评审模式 | Codex + Claude/Gemini 等 | 一个 Provider 写，另一个 Provider 审 | 高 |
| 多 Provider 并行交付模式 | 多个 CLI | 多 worktree 并行实现 + 交叉评审 | 最高 |

UI 不应提示“缺少 Claude/Gemini 所以不能用”，而应提示：

```text
当前检测到：Codex
可用模式：单 Provider 多角色
建议：安装 Claude Code 或 Gemini CLI 后可开启跨 Provider 评审
```

这能降低首次使用门槛，同时保留高阶爆点。

### 6.5 人类门禁

必须人工确认的动作：

- 应用 Reviewer 建议让 Implementer 修改。
- Commit。
- Push。
- 创建 PR。
- 清理 worktree。
- 合并分支。

系统可以推荐动作，但不能替用户做最终责任动作。

## 7. Agent 角色设计

Agent 角色是任务职责，不等同于具体工具。一个 Provider 可以启动多个独立会话来扮演不同角色，但每个角色必须使用独立上下文、独立输出文件和独立日志，避免“自己写完自己随口夸自己”的黑盒感。

推荐配置示例：

```json
{
  "providers": {
    "codex": { "enabled": true, "command": "codex" },
    "claude": { "enabled": false, "command": "claude" }
  },
  "roles": {
    "implementer": { "provider": "codex", "mode": "write" },
    "specReviewer": { "provider": "codex", "mode": "readonly" },
    "codeReviewer": { "provider": "codex", "mode": "readonly" }
  }
}
```

当多个 Provider 可用时，系统应优先推荐交叉组合：

```text
Codex 实现，Claude 评审
Claude 实现，Codex 评审
Gemini 做测试和边界场景复核
```

当只有一个 Provider 可用时，系统应自动降级为单 Provider 多角色模式，并在评审摘要中标注：

```text
评审可信度：中
原因：实现与评审使用同一 Provider，但已使用独立只读会话和不同评审 Prompt。
```

### 7.1 Implementer

负责根据 Spec、Plan、Task 修改代码。

默认能力：

- 读取上下文。
- 修改 worktree 中的文件。
- 运行项目登记的测试命令。
- 生成实现总结。

默认限制：

- 不允许访问 worktree 外部路径。
- 不允许自动 Commit/Push。
- 不允许修改 `.env`、密钥、SSH、系统目录。

### 7.2 Spec Reviewer

检查实现是否满足 Spec 和验收标准。

重点问题：

- 是否遗漏验收标准。
- 是否误解需求。
- 是否做了超出范围的功能。
- 是否破坏已有行为。

### 7.3 Code Reviewer

检查代码质量、架构、可维护性。

重点问题：

- 是否引入重复逻辑。
- 是否破坏本地风格。
- 是否过度抽象。
- 是否存在明显 bug。

### 7.4 Test Reviewer

检查测试覆盖和回归风险。

重点问题：

- 是否有必要测试。
- 是否测试了边界条件。
- 是否存在未运行测试。
- 是否需要补充回归范围。

### 7.5 Security Reviewer

检查安全和敏感操作。

重点问题：

- 是否泄露密钥。
- 是否引入注入风险。
- 是否绕过鉴权。
- 是否修改依赖、脚本、配置、迁移等高风险文件。

## 8. 状态机

### 8.1 Task 状态

```text
backlog
-> ready
-> running
-> implemented
-> reviewing
-> changes_requested
-> approved
-> pr_ready
-> done
```

失败状态：

```text
failed
cancelled
conflicted
blocked
```

### 8.2 Agent Run 状态

```text
queued
-> starting
-> running
-> waiting_approval
-> completed
```

失败状态：

```text
failed
aborted
timeout
```

### 8.3 Review 状态

```text
pending
-> running
-> approve | request_changes | block
```

汇总结论：

```text
all_approved
changes_needed
blocked
human_override
```

## 9. Repo-native 文件结构

建议第一版使用 Markdown + JSON + NDJSON：

```text
.zhiflow/
  config.json
  agents.json
  locales/
    zh-CN.json
    en-US.json
  changes/
    ZF-001-login/
      spec.md
      plan.md
      tasks.md
      task-graph.json
  tasks/
    ZF-001-01-api.md
    ZF-001-02-ui.md
    ZF-001-03-tests.md
  runs/
    run-20260702-001/
      run.json
      events.ndjson
      summary.md
      diff.patch
      test-output.txt
      reviews/
        claude-spec-review.md
        codex-code-review.md
        gemini-test-review.md
      review-summary.md
  worktrees/
    ZF-001-01-api/
    ZF-001-02-ui/
    ZF-001-03-tests/
```

SQLite 可以用于本地 WebUI 的索引和快速查询，但不能成为唯一事实来源。事实来源必须是 `.zhiflow/` 文件，方便 Git 跟踪、人工查看、AI 读取和开源传播。

`config.json` 中需要保存默认语言：

```json
{
  "locale": "zh-CN",
  "fallbackLocale": "en-US"
}
```

项目级 `.zhiflow/locales/` 用于覆盖或扩展内置文案。第一版先支持界面文案和 Agent 提示词模板的国际化，不急着支持用户生成内容自动翻译。

## 10. CLI 命令设计

### 10.1 初始化

```bash
zhiflow init
```

作用：

- 检查是否在 Git 仓库内。
- 创建 `.zhiflow/`。
- 检测可用 Agent CLI。
- 生成默认配置。
- 初始化默认语言配置，默认 `zh-CN`。

可选：

```bash
zhiflow init --lang zh-CN
zhiflow init --lang en-US
```

### 10.2 启动工作台

```bash
zhiflow board
```

或：

```bash
npx zhiflow
```

作用：

- 启动本地服务。
- 打开 WebUI。
- 监听 `.zhiflow/` 状态变化。
- 根据 `.zhiflow/config.json` 加载界面语言。

### 10.3 创建变更

```bash
zhiflow create "Add email/password login"
```

中文示例：

```bash
zhiflow create "新增邮箱密码登录"
```

作用：

- 生成 Spec、Plan、Tasks。
- 创建任务卡。
- 进入 WebUI 待确认。

### 10.4 并行执行

```bash
zhiflow run ZF-001 --parallel 3
```

作用：

- 为每个子任务创建 worktree。
- 按配置选择 Agent。
- 并行启动实现。
- 记录事件和日志。

### 10.5 评审

```bash
zhiflow review ZF-001 --reviewers spec,code,test
```

作用：

- 收集 Diff 和测试输出。
- 启动只读 Reviewer Agent。
- 生成评审报告和汇总结论。

### 10.6 创建 PR

```bash
zhiflow pr ZF-001
```

作用：

- 汇总任务结果。
- 生成 PR 描述。
- 调用 `gh` CLI 创建 PR。
- 保存 PR 元数据。

## 11. WebUI 原型结构

第一版只有一个主工作台，不做营销首页。默认界面语言为简体中文，顶部提供轻量语言切换入口，例如 `中文 / English`。

### 11.1 布局

```text
左侧：项目与变更导航
中间：Kanban + Task Group
右侧：Agent Run / Diff / Review / Decision
底部：并行执行状态条
```

### 11.2 第一屏信息

用户进入后应立刻看到：

- 当前仓库。
- 当前变更。
- 子任务列。
- 哪些 Agent 正在跑。
- 哪些 worktree 正在使用。
- 哪些任务需要 Review。
- 当前是否可以创建 PR。
- 当前界面语言和语言切换入口。

### 11.3 UI 气质

方向：研发工作台、命令中心、精密但不花哨。

视觉关键词：

```text
local-first
terminal-grade
parallel lanes
review gates
git-native
```

避免：

- SaaS 营销页。
- 大面积渐变背景。
- 空泛 AI 聊天框。
- 传统后台表格堆叠。
- 卡片套卡片。

建议配色：

- 背景：深墨色或冷灰黑。
- 主信息区：低对比深色面板。
- 状态色：绿色表示通过，琥珀表示需处理，红色表示阻塞，蓝色表示运行中。
- Git Diff：绿色新增、红色删除，但控制饱和度。

### 11.4 中文优先与国际化体验

第一版所有产品界面默认中文，包括导航、按钮、状态、评审结论、风险提示、空状态、错误提示和确认弹窗。

推荐中文术语：

| 英文概念 | 中文界面文案 |
| --- | --- |
| Spec-to-PR | 从规格到 PR |
| Worktree | 隔离工作区 |
| Agent Run | Agent 执行 |
| Implementer | 实现 Agent |
| Reviewer | 评审 Agent |
| Human Gate | 人工门禁 |
| Request Changes | 要求修改 |
| Create PR | 创建 PR |
| Ready for Review | 待评审 |
| Approved | 已通过 |
| Blocked | 已阻塞 |

国际化原则：

- 所有 UI 文案必须通过 `t(key)` 获取，禁止在组件中硬编码中文或英文。
- 状态枚举保存稳定英文 key，例如 `changes_requested`，展示层再翻译。
- 日期、时间、数字、相对时间使用 locale-aware formatter。
- Agent prompt 模板也需要按语言区分，避免中文用户看到英文评审报告。
- README 和官网文案中英双语，但产品第一屏默认中文。
- 第一版至少内置 `zh-CN` 和 `en-US`。

## 12. 技术架构

推荐采用 TypeScript monorepo。

```text
packages/
  cli/
  core/
  server/
  web/
  agent-adapters/
```

### 12.1 CLI

- Commander 或 Clipanion。
- 负责命令入口、初始化、启动服务。

### 12.2 Core

- `.zhiflow/` 文件读写。
- Task graph。
- 状态机。
- Agent Run 编排。
- Review 汇总。
- Git worktree 管理。

### 12.3 Server

- Fastify 或 Hono。
- 提供本地 REST API。
- SSE 或 WebSocket 推送 Agent 事件。

### 12.4 Web

- React + Vite + TypeScript。
- TanStack Query。
- Zustand。
- Tailwind CSS。
- i18next 或 Lingui。
- dnd-kit。
- xterm.js。
- Monaco 或 CodeMirror Diff。

Web 国际化实现原则：

- `packages/web/src/locales/zh-CN.json`
- `packages/web/src/locales/en-US.json`
- `packages/web/src/i18n.ts`
- 语言设置优先级：URL 参数 -> 用户设置 -> `.zhiflow/config.json` -> 浏览器语言 -> `zh-CN`。
- 组件只使用文案 key，不直接拼接中英文句子。
- 按钮、状态、表格列、弹窗标题、错误提示全部进入文案包。

### 12.5 Agent Adapters

第一版适配：

- Codex CLI。
- Claude Code。
- Gemini CLI。

每个 Adapter 提供统一接口：

```text
detect()
buildCommand(context)
start()
streamEvents()
abort()
summarize()
```

## 13. 安全与权限原则

默认安全模式：

- Agent 只能在绑定 worktree 内执行。
- Reviewer 默认只读。
- Commit、Push、PR 需要人工确认。
- 高风险文件修改必须突出显示。
- 网络访问、依赖安装、迁移脚本、删除文件标为中高风险。

高风险路径：

```text
.env
*.pem
*.key
id_rsa
package-lock.json
pnpm-lock.yaml
yarn.lock
docker-compose.yml
Dockerfile
database migrations
```

第一版不追求强沙箱，但要在 UI 和日志中清楚展示风险，并保留事件记录。

## 14. 冲 1k star 的发布策略

### 14.1 README 第一屏

必须包含：

- 一句话定位。
- 10 秒 GIF。
- 一条启动命令。
- 支持的 Agent 图标或名称。

- 多 Agent 并行编码 / Parallel AI coding agents。
- Git worktree 隔离 / Git worktree isolation。
- 交叉 Agent 评审 / Cross-agent review。
- 一键创建 PR / One-click PR。

其他能力都放到后面，不抢主叙事。

### 14.3 首发 Demo

建议准备两个 Demo：

- 小型前端功能：最容易做视觉演示。
- 后端 API + 测试：最容易展示 Spec、测试和 Review 价值。

## 16. 风险与应对

### 16.1 Agent CLI 差异大

应对：

- 第一版先做命令适配和 stdout 采集。
- 不强依赖结构化事件。
- 能解析就解析，不能解析就保留原始日志。

### 16.2 多 Agent 并行容易冲突

应对：

- 默认按文件范围拆任务。
- 运行前提示潜在冲突。
- 同一文件多任务修改时进入 `conflicted`。
- 第一版不自动复杂合并。

### 16.3 Review 结果质量不稳定

应对：

- Reviewer 必须引用 Diff 和 Spec。
- 输出结构化结论。
- 高风险发现必须给出文件和原因。
- 最终由人类决定。

### 16.4 范围容易膨胀

应对：

- 所有功能围绕 Demo 路径判断。
- 不服务 Demo 的功能延后。
- 不做 SaaS，不做团队权限，不做发布管理。

## 17. 当前结论

智流原始大平台愿景可以保留，但开源冲星路线应先做智流 Lite。

第一版不是项目管理系统，而是：

```text
多 Agent 编码流水线 + Git worktree 隔离 + 交叉评审 + 人类 PR 门禁
```

这条路线足够尖，传播句足够短，Demo 足够强，也能自然演进到后续完整智流平台。
