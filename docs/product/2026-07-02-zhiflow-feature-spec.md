# 智流功能文档 V0.1

## 1. 产品定位

智流是一款面向项目开发组的 AI-native 项目全生命周期管理平台。

它的目标不是单独替代项目管理、文档管理、知识库、测试管理或 AI 编码工具，而是把这些环节串成一条连续的项目工作流：

```text
需求输入 -> 知识沉淀 -> Spec 驱动 -> 任务落地 -> 开发执行 -> 测试提测 -> Bug 修复 -> 发布复盘
```

智流需要同时服务项目经理、产品经理、研发经理、研发人员、测试人员、运维/交付人员等角色，并允许 AI 协作者参与开发和分析工作。

## 2. 产品形态

### 2.1 Web 端

Web 端是团队协作入口，提供完整的项目管理、知识库、Spec/PRD 视图、任务、Bug、测试、发布和成员协作能力。

适合场景：

- 产品、项目经理、测试、管理者日常使用。
- 开发者在没有安装客户端时查看任务、文档和协作信息。
- 团队成员通过浏览器快速访问项目空间。

### 2.2 桌面客户端

桌面客户端是增强版主工作台，功能原则上覆盖 Web 端核心能力，并额外提供本地开发能力。

第一版确定将桌面客户端设计为主入口之一。对研发人员和研发经理来说，桌面客户端应是首选入口；对产品经理、项目经理、测试人员等角色来说，可以根据习惯选择 Web 端或客户端。

增强能力包括：

- 连接本地代码仓库。
- 调用 Codex、Claude Code 等 AI 编码工具。
- 读取本地分支、变更、Diff 和测试结果。
- 执行命令、运行测试、生成提交说明。
- 将开发执行结果回写到任务、Bug、提测和项目记录中。

### 2.3 移动 App

移动 App 暂定为后续版本能力，主要用于通知、评论、审批、查看进度、处理紧急 Bug 和轻量协作。

第一阶段不作为核心生产力入口。

## 3. 核心角色

| 角色 | 核心诉求 |
| --- | --- |
| 项目经理 | 管进度、排优先级、协调资源、识别风险、推进交付 |
| 产品经理 | 上传需求、维护产品 Spec/PRD 视图、确认验收标准 |
| 研发经理 | 评审技术 Plan、拆解研发任务、管理代码质量和技术风险 |
| 研发人员 | 接收任务、理解上下文、调用 AI 编码、提交变更 |
| 测试人员 | 生成测试用例、执行提测、提交 Bug、验收修复 |
| 运维/交付人员 | 管理版本、部署记录、发布说明和回滚信息 |
| 干系人/访客 | 查看项目进展、参与评论和指定评审 |
| AI 协作者 | 根据任务上下文参与分析、开发、测试和总结 |

第一版权限应尽量和项目团队角色一致，而不是抽象成单一的管理员/成员模型。

推荐项目团队角色：

- 项目经理：对项目进度、范围、优先级、资源协调和发布节奏负责。
- 产品经理：对需求、产品 Spec、验收标准和用户价值负责。
- 研发经理：对技术 Plan、研发任务拆解、代码质量、架构风险和高风险技术决策负责。
- 研发人员：对任务实现、代码变更、自测和 PR/MR 负责。
- 测试人员：对测试用例、提测验证、Bug 质量和测试结论负责。
- 运维/交付人员：对发布、部署、回滚和环境记录负责。
- 干系人/访客：查看项目进展、评论和参与指定评审，不参与执行操作。
- AI 协作者：由人类成员授权后参与分析、生成、开发、测试和总结，不作为最终责任人。

## 4. 一级功能模块

### 4.1 工作区与项目空间

用于承载工作区、项目和成员协作。

产品第一版确定以“项目”为核心入口。用户进入智流后，主要围绕具体项目开展工作，而不是先进入分散的全局需求中心、任务中心或知识库中心。

推荐信息层级：

```text
工作区
-> 项目
-> 项目内模块：总览、需求、知识库、Spec、任务、Bug、测试、代码、发布
```

这种结构的好处是每个功能都天然带有项目上下文，AI 在生成 Spec、拆解任务、执行开发、分析 Bug 时，也能直接获取当前项目的完整背景。

初步功能：

- 工作区管理。
- 项目创建与归档。
- 项目成员与角色权限。
- 项目总览仪表盘。
- 项目动态时间线。
- 项目风险与待办汇总。

第一版不做完整组织架构，不引入公司、部门、团队、小组等复杂层级。

推荐第一版层级：

```text
工作区：一个公司、团队、项目组或个人空间
项目：实际开发和交付单元
成员：加入工作区或项目的人
角色：按照项目团队职责分配
```

权限设计第一版应保持简单，优先满足项目协作，不追求企业级组织管控。权限不按完整组织架构划分，而按项目团队职责和关键动作划分。

工作区与项目关系：

- 工作区是协作空间。
- 项目是工作区里的具体交付单元。
- 一个用户可以加入多个工作区。
- 一个工作区可以有多个项目。
- 一个项目只能属于一个工作区。
- 项目成员必须来自工作区成员。

工作区负责：

- 成员池。
- 项目列表。
- 工作区级配置。
- 对象存储配置。
- 模型配置。
- GitHub/GitLab 集成配置。
- 后续计费/套餐。

项目负责：

- 项目成员和项目角色。
- 需求/变更。
- Spec。
- 任务 Kanban。
- Bug。
- 提测。
- 发布。
- 项目知识库。
- 代码仓库绑定。
- AI 执行记录。

创建与管理规则：

- 创建工作区的人自动成为工作区所有者。
- 工作区所有者/管理员可以创建项目。
- 创建项目的人默认成为项目经理。
- 创建项目时可以指定其他工作区成员为项目经理。
- 项目经理可以邀请工作区成员进入项目，并分配项目角色。
- 工作区所有者/管理员可以更换项目经理、归档项目或处理权限问题。

关键动作权限建议：

| 动作 | 建议默认权限 |
| --- | --- |
| 创建项目 | 工作区管理员/项目经理 |
| 邀请项目成员 | 项目经理 |
| 配置 Kanban 流程 | 项目经理 |
| 创建变更 | 项目成员 |
| 确认 Proposal | 产品经理/项目经理 |
| 确认 Spec | 产品经理/研发经理 |
| 确认 Plan | 研发经理 |
| 确认 Tasks | 项目经理/研发经理 |
| 调用 AI 开发 | 研发人员/研发经理 |
| 批准中风险 AI 操作 | 当前任务负责人/研发经理 |
| 批准高风险 AI 操作 | 研发经理/项目经理 |
| Commit/Push/创建 PR/MR | 当前任务负责人/研发人员 |
| 发起提测 | 研发人员/任务负责人 |
| 通过测试 | 测试人员 |
| 关闭 Bug | 测试人员/创建人/项目经理 |
| 创建轻量版本发布 | 项目经理/运维或交付人员 |
| 创建大版本基线发布 | 项目经理/研发经理 |
| 归档变更到基线 Spec | 项目经理/研发经理 |

### 4.2 项目总览 Dashboard

项目总览是用户进入项目后的第一屏，用于快速判断项目当前进展、待处理事项和风险点。

设计原则：

- 面向项目组日常协作，不做花哨管理大屏。
- 优先展示需要行动的信息。
- 同时覆盖 SDD 进展、任务执行、测试 Bug、AI 执行和发布状态。

建议区域：

- 项目状态。
- 变更进度。
- 任务进度。
- 测试/Bug 状态。
- AI 执行状态。
- 发布状态。
- 风险与待办。
- 最近动态。

核心问题：

- 项目现在进展如何？
- 接下来谁该做什么？
- 有什么风险需要马上处理？

初步指标：

- 进行中的变更数量。
- 待澄清变更。
- 规格评审中变更。
- 待拆任务变更。
- 开发中变更。
- 验证中变更。
- 待开发任务。
- 开发中任务。
- 待提测任务。
- 测试中任务。
- 阻塞任务。
- 新增 Bug。
- 高优先级 Bug。
- 待验证 Bug。
- 正在执行的 AI 会话。
- 等待人工确认的 Diff。
- 执行失败的 AI 任务。
- 最近发布。
- 未归档变更。

### 4.3 需求管理

用于收集、整理和跟踪原始需求。

需求管理主要承载原始输入，不直接等同于开发任务。需求经过确认后，可以创建为一次 SDD 变更。

初步功能：

- 需求文档上传。
- 需求手动录入。
- 需求来源标记。
- 需求优先级。
- 需求状态流转。
- 需求讨论与评论。
- AI 需求摘要。
- AI 需求冲突识别。

### 4.4 变更管理

用于承接一次可讨论、可评审、可实现、可归档的项目变化。变更是智流 SDD 流程的核心入口。

变更来源：

- 需求文档上传。
- 手动录入的新功能。
- Bug 修复。
- 体验优化。
- 技术债治理。
- 重构。
- 安全/性能/兼容性改造。
- 项目讨论中沉淀出来的行动项。

创建变更页面第一版建议包含：

- 变更标题。
- 变更类型：新功能、需求调整、Bug 修复、优化、重构、技术债、其他。
- 来源材料：上传文档、选择知识库条目、关联已有需求、关联 Bug、粘贴文本。
- 背景说明：为什么要做。
- 目标结果：希望达成什么。
- 初始范围：包含什么、不包含什么。
- 优先级。
- 期望完成时间。
- 负责人或提出人。
- 关联模块或代码仓库。
- 是否让 AI 先探索/澄清。

创建后的变更应进入 SDD 工作流，而不是直接进入 Kanban。

推荐状态：

```text
草稿 -> 澄清中 -> 待生成规格 -> 规格评审中 -> 待拆任务 -> 开发中 -> 验证中 -> 已归档
```

这些状态描述的是一次变更的生命周期，Kanban 状态描述的是任务执行状态，两者需要区分。

可以提供“变更看板”查看 SDD 进展，但它和“任务 Kanban”应是两个不同视图：

- 变更看板：展示每个变更在 SDD 流程中的位置。
- 任务 Kanban：展示从 Tasks 拆出来的具体执行任务。

### 4.5 项目知识库

用于存放和检索项目相关资料，让团队和 AI 都能获得稳定上下文。

第一版知识库建议分为三类：

- 项目资料库：需求文档、会议纪要、业务规则、竞品资料、接口说明等。
- 项目基线 Spec：当前系统能力、业务规则、已归档变更和长期有效的规格。
- 代码知识库：仓库结构、模块说明、关键入口、接口位置、数据模型、技术栈、测试命令和构建命令。

代码知识库第一版纳入核心范围。它是桌面客户端的重要增强能力：客户端负责扫描本地代码仓库并生成代码上下文，Web 端可以查看和引用扫描结果。

初步功能：

- 文档上传。
- Markdown/富文本内容维护。
- 接口文档沉淀。
- 业务规则沉淀。
- 代码库说明沉淀。
- 知识条目标签。
- 全文检索。
- 可选语义推荐。
- 引用溯源。
- AI 问答。

第一版建议支持的资料格式：

- Markdown。
- TXT。
- PDF。
- DOCX。
- XLSX。
- 图片附件。
- 网页链接。
- 代码仓库扫描结果。

代码知识库初步功能：

- 连接本地仓库。
- 识别技术栈。
- 识别项目结构。
- 识别关键入口文件。
- 识别路由/API/Controller。
- 识别数据模型/Schema。
- 识别测试命令。
- 识别构建命令。
- 生成模块说明。
- 支持手动修正代码知识。
- 为 Plan、Tasks 和 AI 开发执行提供上下文。

V0.1 代码知识库定位：

```text
开发上下文摘要，不是完整代码搜索引擎
```

生成方式：

```text
桌面客户端扫描本地仓库
-> 生成 code_knowledge.json
-> 用户预览/确认
-> 上传到项目代码知识库
-> 用于 Plan、Tasks 和 AI 开发上下文装配
```

扫描内容：

- 仓库名称。
- 远程仓库地址。
- 默认分支。
- 技术栈。
- 包管理器。
- 项目结构。
- 主要模块。
- 关键入口文件。
- 路由/API/Controller 位置。
- 数据模型/Schema/Migration 位置。
- 启动命令。
- 测试命令。
- 构建命令。
- 重要配置文件。
- README/项目文档摘要。
- 代码规范或约束。
- 禁止修改路径。

扫描来源：

- README。
- package.json。
- pom.xml。
- build.gradle。
- go.mod。
- 目录结构。
- 配置文件。
- 路由/Controller。
- 数据库 migration/schema。
- 测试配置。

确认原则：

- 自动扫描结果需要用户预览。
- 用户可以手动修正技术栈、模块说明、测试命令和构建命令。
- 确认后的代码知识才进入上下文包。

V0.1 暂不做：

- 深度 AST 分析。
- 完整调用链分析。
- 全量代码向量化。
- 复杂依赖图。
- 自动代码搜索引擎。

### 4.6 Context Engineering 上下文工程

用于把项目中的需求、Spec、代码知识、历史变更、Bug、测试结论和发布记录装配成 AI 可用、可追溯、可控制的上下文包。

智流 V0.1 的目标不是做一个完整 RAG 平台，而是做面向项目交付的轻量上下文工程系统。RAG、向量检索、Embedding 可以作为上下文推荐能力存在，但不应成为第一版主链路的复杂依赖。

核心原则：

- 源文档和已确认 Spec 是事实底座。
- 上下文包是 AI 执行的输入边界。
- 自动化优先依赖结构化关系、项目状态、Spec 关联和全文检索。
- 语义检索只作为推荐和补充，不默认把召回结果直接注入开发上下文。
- 所有进入 Proposal、Spec、Plan、Tasks 和 AI 开发的上下文都要可追溯来源。
- 低可信或未确认内容需要标记，不能直接作为强事实依据。

V0.1 上下文装配来源：

- 当前变更的原始需求和附件。
- 当前变更已生成的 Proposal、Spec、Plan、Tasks。
- 项目基线 Spec。
- 人工确认的业务规则和知识条目。
- 关联模块的代码知识摘要。
- 关联历史变更。
- 关联 Bug 和测试结论。
- 项目配置的技术约束和开发规范。

V0.1 上下文装配方式：

- 显式关联：变更关联的文档、Spec 条目、任务、Bug、仓库和模块。
- 规则装配：按项目、模块、标签、变更类型、任务类型自动选择上下文。
- 全文搜索：从源文档和 Markdown 解析结果中查找关键词相关内容。
- 语义推荐：可选使用 pgvector 推荐可能相关片段，但需要展示来源和可信度。
- 人工确认：高风险变更或关键开发任务可以要求确认上下文包。

上下文包建议包含：

```text
变更摘要
关联 Proposal/Spec/Plan/Tasks
验收标准
相关业务规则
相关代码知识
相关历史 Bug
测试重点
排除范围
风险提示
引用来源
```

这套机制的目标是减少人工整理上下文，而不是让系统变成不可控的自动 RAG。

### 4.7 Spec 中心

用于管理项目的规格资产，让需求、方案、任务、测试和 AI 开发都围绕同一套可追踪的 Spec 展开。

智流中的 PRD 不再被理解为一个孤立文档，而是 Spec 的“产品视图”。同一份变更可以同时拥有产品视图、技术视图、任务视图和测试视图。

推荐 Spec 类型：

- 项目基线 Spec：描述当前系统已经具备的能力、业务规则、接口、数据约束和重要技术决策。
- 变更 Spec：描述一次新增功能、需求调整、Bug 修复或重构的目标、范围和影响。
- 产品视图：说明为什么做、做什么、用户场景、验收标准，可导出为 PRD。
- 技术视图：说明影响范围、模块设计、接口、数据、边界场景和实现约束。
- 任务视图：说明可执行任务、依赖关系、完成标准和测试点。
- 测试视图：说明测试用例、验收路径、回归范围和风险点。

初步功能：

- 需求资料生成 Spec 草稿。
- PRD 视图生成与编辑。
- Spec 版本管理。
- Spec 差异对比。
- Spec 评审与确认。
- Spec 条目与任务、Bug、测试用例、代码变更关联。
- 完成后将变更 Spec 归档并合并到项目基线 Spec。

### 4.8 SDD：Spec Driven Development 工作流

用于通过规格驱动开发，把需求、产品规格、设计约束、任务、测试和 AI 执行连接起来。

SDD 在智流中指 Spec Driven Development，不是 Software Design Document。它不是单独生成一份传统技术设计文档，而是让 Spec 成为项目开发过程的中心依据。

参考 SpecKit 和 OpenSpec 的常见做法，智流第一版建议采用“变更驱动”的 SDD 流程：

```text
创建变更
-> 探索/澄清
-> 生成 Proposal
-> 生成 Spec
-> 生成 Plan
-> 生成 Tasks
-> 一致性检查
-> 人工评审
-> AI/人工实现
-> 验证/收敛
-> 归档并更新基线 Spec
```

进入开发 Kanban 的前置条件：

- 必须已经生成 Proposal。
- 必须已经生成 Spec。
- 必须已经生成 Plan。
- 必须已经生成 Tasks。
- 至少完成项目设定要求的评审动作。

也就是说，原始需求、Proposal、Spec 和 Plan 本身不直接进入开发 Kanban。只有从 Tasks 中拆出的可执行任务，才进入 Kanban。

各阶段含义：

- 创建变更：从需求上传、手动录入、Bug、优化建议或项目讨论中创建一次变更。
- 探索/澄清：AI 帮助识别需求缺口、冲突、边界和待确认问题。
- Proposal：说明为什么做、改什么、不改什么、范围和成功标准。
- Spec：沉淀功能需求、场景、验收标准和约束。
- Plan：沉淀技术方案、影响范围、接口/数据/模块变化和实现策略。
- Tasks：把 Plan 拆为可执行任务，并进入 Kanban。
- 一致性检查：检查 Proposal、Spec、Plan、Tasks 是否冲突或遗漏。
- 人工评审：产品经理、项目经理、研发经理按角色确认。
- AI/人工实现：开发人员或 AI 协作者根据任务和 Spec 执行。
- 验证/收敛：对照 Spec 和 Tasks 检查是否完成，补齐遗漏任务。
- 归档：完成后将变更沉淀到项目基线 Spec 和知识库。

第一版可以把这些阶段产品化为按钮和状态，而不是暴露命令行概念。

SDD 流程适合放在“变更详情页”中呈现，例如使用阶段条、时间线、文档标签页和评审状态。它不建议直接作为开发 Kanban 的列。

### 4.9 SDD 评审流

用于确认 Proposal、Spec、Plan 和 Tasks 的质量与责任归属。

第一版采用轻量评审，不做复杂审批流。

推荐默认评审责任：

| 产物 | 默认评审角色 | 评审重点 |
| --- | --- | --- |
| Proposal | 产品经理、项目经理 | 方向、范围、价值、优先级是否正确 |
| Spec | 产品经理、研发经理 | 需求、场景、验收标准、约束是否清晰 |
| Plan | 研发经理 | 技术方案、影响范围、风险和实现策略是否合理 |
| Tasks | 项目经理、研发经理 | 任务是否可执行、可排期、可验证 |

评审状态：

```text
草稿 -> 待评审 -> 需修改 -> 已确认
```

评审操作：

- 通过。
- 退回修改。
- 评论。
- 提及成员。
- 查看 AI 生成依据。
- 查看与上一版本的差异。
- 查看关联资料和引用来源。

评审简化规则：

- 普通变更允许项目经理按项目配置跳过部分评审。
- 小型 Bug 或文案调整可以简化 Proposal、Spec 或 Plan。
- 高风险变更不允许跳过 Plan 和 Tasks 评审。
- 涉及权限、安全、支付、数据结构、核心业务流程的变更必须完整评审。

完成评审后，Tasks 才能进入任务 Kanban。

### 4.10 任务 Kanban

用于承载从 SDD Tasks 生成出来的执行任务。Kanban 是执行视图，不是规格编写视图。

推荐默认状态：

```text
待开发 -> 开发中 -> 待提测 -> 测试中 -> 待发布 -> 已完成
```

“待拆解”属于 SDD 流程中的 Tasks 生成与任务细化阶段，不建议作为开发 Kanban 的默认列。若项目经理希望保留，可以作为项目配置项开启。

第一版提供上述默认流程，同时允许项目经理在项目维度进行配置。

可配置方向：

- 是否启用某些状态，例如“测试中”“待发布”。
- 是否允许任务跳转状态。
- 是否要求进入某个状态前填写必要信息。
- 是否绑定自动动作，例如进入“待提测”时生成提测单。
- 是否限制某些角色操作状态流转。

第一版不建议做完全自由的自定义工作流，避免流程统计、权限控制和 AI 自动化规则过早复杂化。

初步功能：

- 从 SDD Tasks 生成任务卡片。
- 手动创建任务。
- 任务负责人。
- 优先级和截止时间。
- 子任务。
- 任务依赖。
- 任务评论。
- 任务与需求、Spec/PRD 视图、Bug、代码变更关联。
- 任务与 Spec 条目关联。

任务卡片建议展示：

- 任务标题。
- 任务类型：功能、Bug 修复、优化、重构、技术债、测试。
- 所属变更。
- 关联 Spec 条目。
- 负责人。
- AI 协作者。
- 优先级。
- 截止时间。
- 当前分支。
- PR/MR 状态。
- 提测状态。
- 阻塞状态。

Kanban 应支持按变更、负责人、优先级、任务类型、版本、AI 执行状态筛选，也支持按所属变更分组展示。

### 4.11 AI 开发执行

用于把任务交给 Codex、Claude Code 或其他 AI Agent 执行。

该模块是桌面客户端相对 Web 端的核心增强能力。Web 端可以查看 AI 执行记录、任务状态和代码变更摘要，但真正涉及本地仓库、命令执行、测试运行、Diff 检查和分支操作的能力，第一版优先在桌面客户端实现。

AI 在智流中定位为“特殊协作者/执行器”，不是单纯的功能按钮，也不是强人格化的虚拟员工。

设计原则：

- 每个任务必须有人类负责人。
- 每个任务可以绑定一个或多个 AI 协作者。
- AI 可以参与分析、生成、开发、测试和总结。
- AI 的执行过程需要留下记录。
- AI 生成的代码、文档、任务和测试结论需要人工确认。
- 任务责任仍归属人类负责人，AI 不作为最终责任人。

示例：

```text
任务：实现登录接口
负责人：张三
AI 协作：Codex
AI 当前动作：正在修改登录模块
执行记录：
- 已读取 Spec
- 已分析 user 表结构
- 已修改 LoginService
- 已运行单元测试
- 等待张三确认 Diff
```

初步功能：

- 任务上下文打包。
- 选择本地代码仓库。
- 选择分支或创建任务分支。
- 基于 Git worktree 创建隔离开发目录。
- 选择 AI 执行器。
- AI 开发会话记录。
- 本地命令执行记录。
- 测试结果采集。
- Diff 预览。
- 提交信息生成。
- PR/MR 创建。
- 任务状态自动回写。

第一版采用半自动执行模式，并设置人工确认点。

AI 可以自动完成：

- 分析任务上下文。
- 阅读 Spec、Plan、Tasks 和代码知识库。
- 在隔离 worktree 中修改代码。
- 执行允许的命令。
- 运行测试和构建。
- 总结变更和风险。

必须人工确认后才能执行：

- 应用高风险命令。
- 提交 Commit。
- Push 到远程仓库。
- 创建 PR/MR。
- 将任务推进到提测或完成。

推荐执行流程：

```text
选择任务
-> 选择仓库
-> 创建任务分支
-> 创建 Git worktree
-> 选择 AI 执行器
-> 打包上下文
-> AI 在 worktree 中开发
-> 展示执行日志和 Diff
-> 运行测试/构建
-> 人工确认
-> Commit
-> Push
-> 创建 PR/MR
-> 回写任务状态
```

Worktree 设计原则：

- 每个任务默认使用独立 worktree。
- worktree 目录由客户端管理。
- 不直接污染用户当前工作目录。
- 支持并行处理多个任务。
- 支持任务完成后清理 worktree。
- 支持异常中断后恢复执行现场。
- 支持查看 worktree 与目标分支的 Diff。

AI 执行器接入原则：

- 第一版不自研替代 Codex、Claude Code 等原生编码 Agent。
- 智流客户端优先调用用户本机已安装的原生 CLI/Agent。
- 智流负责上下文打包、worktree 隔离、进程编排、权限确认、日志采集、Diff 汇总和状态回写。
- UI 上统一抽象为“AI 执行器”，底层可以接 Codex、Claude Code 或其他工具。

原生执行记录采集方式：

- 通过客户端启动和托管 Agent 进程，采集 stdout/stderr、退出码和运行时长。
- 将每次执行包装为一个 AI 开发会话，关联任务、变更、Spec、仓库、分支和 worktree。
- 在执行前后采集 Git 状态和 Diff，记录文件新增、修改、删除。
- 记录 Agent 触发的命令、命令结果、测试输出和构建输出。
- 将关键事件结构化展示，例如读取上下文、修改文件、运行测试、等待确认、生成提交说明。
- 保留原始终端日志，供开发人员排查。
- 对支持结构化输出的 Agent，优先读取 JSON/event stream；对只支持终端输出的 Agent，先做终端日志采集和事件抽取。

推荐执行记录结构：

```text
AI 开发会话
-> 任务与变更信息
-> 使用的执行器
-> 仓库/分支/worktree
-> 上下文包版本
-> 终端日志
-> 命令记录
-> 文件变更
-> 测试/构建结果
-> 人工确认记录
-> Commit/PR/MR 结果
```

权限与安全控制：

第一版需要提供类似 Codex 权限模式的开关，用于控制 AI 执行器在本地开发时的自动化程度。

推荐三档模式：

- 请求批准：所有文件修改、命令执行、网络访问、Git 操作都需要用户确认。
- 智能审批：低风险操作自动执行，中高风险操作弹窗确认。建议作为默认模式。
- 完全访问权限：允许 AI 在授权仓库和 worktree 内自动执行大部分操作，但 Commit、Push、创建 PR/MR、推进任务状态仍需要人工确认。

风险分级建议：

- 低风险：读取文件、查看 Git 状态、运行项目配置中登记的测试/构建命令、生成摘要。
- 中风险：修改普通业务代码、安装依赖、访问互联网、执行未登记命令、批量格式化文件。
- 高风险：删除文件、修改配置/密钥文件、修改依赖锁文件、数据库迁移、执行 shell 管道脚本、强推、清理 Git 历史、访问 worktree 外部路径。

弹窗确认内容应包括：

- AI 准备执行的操作。
- 风险等级。
- 涉及的文件或命令。
- 为什么需要执行。
- 允许一次、始终允许同类操作、拒绝。

安全边界：

- 默认只允许 AI 访问当前任务绑定的 worktree。
- 访问 worktree 外部文件必须确认。
- 访问互联网必须按权限模式确认。
- `.env`、密钥、证书、SSH Key、系统目录等敏感路径默认受保护。
- 中高风险操作需要弹窗确认，除非用户在权限模式或项目策略中明确放宽。

### 4.12 代码仓库与 Git 集成

用于连接项目代码仓库，并把任务、AI 开发、分支、提交、PR/MR 和测试结果串起来。

第一版确定支持本地 Git 仓库，并直接支持 GitHub/GitLab 的 PR/MR 创建与状态同步。

初步功能：

- 项目绑定一个或多个代码仓库。
- 桌面客户端选择本地仓库路径。
- 识别远程仓库地址。
- 绑定 GitHub/GitLab 账号或访问令牌。
- 读取默认分支和当前分支。
- 基于任务创建开发分支。
- 基于任务创建 Git worktree。
- 生成分支命名建议。
- 查看本地变更 Diff。
- 运行测试和构建命令。
- 生成 Commit Message。
- 提交 Commit。
- Push 到远程仓库。
- 创建 GitHub PR 或 GitLab MR。
- 同步 PR/MR 状态到任务卡。
- 关联 PR/MR、Commit、任务、变更和 Spec 条目。
- PR/MR 合并后自动推进任务状态。

推荐任务分支命名：

```text
feature/<change-key>-<task-key>-short-title
bugfix/<change-key>-<task-key>-short-title
refactor/<change-key>-<task-key>-short-title
```

第一版不强制支持复杂 Git Flow，但需要支持最常见的任务分支开发：

```text
main/develop
-> task branch
-> commit
-> push
-> PR/MR
-> review
-> merge
```

服务端与桌面客户端边界：

服务端负责：

- 仓库绑定。
- GitHub/GitLab OAuth 或 Token 管理。
- Webhook 接收。
- PR/MR 状态同步。
- Commit/PR/MR 元数据保存。
- 任务和 PR/MR 关联。
- 权限校验。
- 发布和归档时引用代码变更记录。

桌面客户端负责：

- 选择本地仓库路径。
- 校验本地仓库远程地址是否匹配项目绑定仓库。
- 创建 Git worktree。
- 创建/切换任务分支。
- 调用 Codex/Claude Code。
- 运行命令和测试。
- 采集终端日志。
- 采集 Diff。
- Commit/Push/创建 PR/MR 前进行人工确认。

边界原则：

- 后端不直接操作用户本地代码。
- 本地文件系统访问只发生在桌面客户端。
- 代码执行、命令执行、测试执行只发生在授权 worktree 中。
- 后端只保存任务、仓库、分支、Commit、PR/MR、Diff 摘要、测试结果和 AI 执行记录。

### 4.13 测试与提测

用于管理开发完成后的测试流程。

第一版采用轻提测单设计。任务从开发进入测试时，必须发起提测，并绑定对应 PR/MR；部署版本或构建记录在第一版不强制，但如果项目已配置 CI/环境集成，应支持关联。

初步功能：

- 发起提测。
- 提测单。
- 关联 PR/MR。
- 关联 Spec 条目。
- 测试范围说明。
- AI 生成测试用例。
- AI 生成提测说明。
- AI 生成回归范围。
- 测试执行记录。
- 测试结论。
- 退回开发。
- 通过验收。

提测单建议包含：

- 提测标题。
- 关联任务。
- 关联变更。
- 关联 Spec 条目。
- PR/MR 链接。
- 变更范围。
- 影响模块。
- 测试重点。
- 自测结果。
- 已知风险。
- 构建/部署信息。
- 测试负责人/测试人员。

推荐流程：

```text
开发中
-> 发起提测
-> 生成提测单
-> 绑定 PR/MR
-> 待提测
-> 测试中
-> 测试通过
-> 待发布/已完成
```

测试失败流程：

```text
测试中
-> 提交 Bug
-> Bug 关联原任务、变更和 Spec
-> 任务退回开发中
-> 修复后重新提测
```

### 4.14 Bug 管理

用于管理缺陷生命周期。

Bug 需要和 SDD、任务、提测流程打通。第一版支持两种处理方式：

- 普通 Bug：关联原任务、提测单和 PR/MR，修复后重新验证关闭。
- 复杂 Bug：如果涉及需求理解、业务规则、架构设计或较大影响范围，可以升级为一次 SDD 变更，重新生成 Proposal、Spec、Plan 和 Tasks。

初步状态：

```text
新建 -> 已确认 -> 修复中 -> 待验证 -> 已关闭 -> 重新打开
```

初步功能：

- Bug 创建。
- 严重程度。
- 优先级。
- 复现步骤。
- 截图/附件。
- 关联任务/需求/版本。
- 关联变更。
- 关联 Spec 条目。
- 关联提测单。
- 关联 PR/MR。
- 指派开发。
- 修复记录。
- 验证记录。

Bug 升级为 SDD 变更的触发条件建议：

- 影响原有需求或验收标准。
- 暴露业务规则不清晰。
- 涉及多个模块。
- 需要调整接口或数据结构。
- 修复方案存在明显技术决策。
- 需要补充回归测试范围。

推荐普通 Bug 流程：

```text
测试提交 Bug
-> 指派开发
-> 修复中
-> 绑定修复 PR/MR
-> 待验证
-> 验证通过
-> 已关闭
```

推荐复杂 Bug 流程：

```text
测试提交 Bug
-> 判断为复杂 Bug
-> 升级为 SDD 变更
-> 生成 Proposal/Spec/Plan/Tasks
-> 进入任务 Kanban
-> 修复并验证
-> 归档变更
```

### 4.15 发布与复盘

用于管理版本发布和项目经验回流。

第一版需要区分两类发布：

- 轻量版本发布：面向日常迭代、小版本上线或内部交付，重点记录本次包含哪些变更、测试结论、风险和发布说明。
- 大版本基线发布：面向重要版本、里程碑或正式对外发布，除了发布记录外，还需要形成新的项目基线，更新基线 Spec、知识库和版本快照。

初步功能：

- 版本计划。
- 发布清单。
- 轻量版本发布。
- 大版本基线发布。
- 变更记录。
- 发布说明生成。
- 回滚方案。
- 发布后问题记录。
- 项目复盘。
- 经验自动沉淀到知识库。

轻量版本发布建议包含：

- 版本号或发布批次。
- 包含的已完成变更。
- 关联 PR/MR。
- 测试结论。
- 已知风险。
- 发布说明。
- 发布人。
- 发布时间。
- 发布后问题记录。

大版本基线发布建议包含：

- 基线版本号。
- 里程碑说明。
- 包含的变更集合。
- 完整发布说明。
- 基线 Spec 快照。
- 关键业务能力清单。
- 关键接口/数据变化。
- 测试报告摘要。
- 风险与回滚方案。
- 发布复盘。
- 后续待办。

推荐流程：

```text
变更完成
-> 变更归档并更新项目基线 Spec
-> 选择已完成变更
-> 创建轻量版本发布
-> 如达到里程碑，升级/创建大版本基线发布
-> 生成基线快照和复盘记录
```

### 4.16 通知与协作

用于保障团队信息同步。

第一版需要提供“我的待办”和通知中心，在 Web 端和桌面客户端都可访问。项目总览回答“项目发生了什么”，我的待办回答“我现在该处理什么”。

初步功能：

- 站内通知。
- 评论与提及。
- 任务状态通知。
- Bug 指派通知。
- 提测通知。
- AI 执行完成通知。
- 邮件/企业微信/飞书/Slack 集成。

我的待办建议包含：

- 待评审 Proposal。
- 待评审 Spec。
- 待评审 Plan。
- 待确认 Tasks。
- 待确认 AI Diff。
- 待批准中高风险命令。
- 待处理任务。
- 待合并 PR/MR。
- 待测试提测单。
- 待验证 Bug。
- 待处理评论或提及。
- 逾期任务。

通知类型建议：

- 分配给我的任务。
- 指派给我的 Bug。
- 需要我评审的变更。
- 需要我确认的 AI 操作。
- AI 执行成功/失败。
- PR/MR 状态变化。
- 提测单状态变化。
- 测试结论变化。
- 发布创建或完成。
- 评论、提及和回复。

待办处理原则：

- 待办必须能跳转到对应上下文，例如变更、任务、Diff、提测单、Bug 或 PR/MR。
- 待办应区分“需要我操作”和“仅通知我”。
- 待办完成后应自动从列表移除或归档。
- 高风险待办应在客户端突出显示，例如 AI 权限确认、失败任务、阻塞任务。

### 4.17 审计日志与操作记录

用于记录项目过程中的关键动作，保障 AI 自动化、项目流转和成员协作可追溯。

V0.1 不做企业级审计系统，但必须提供基础操作留痕。

操作记录类型：

- 业务操作记录。
- AI 执行记录。
- 系统事件记录。

业务操作记录包括：

- 创建变更。
- 生成和确认 Proposal。
- 生成和确认 Spec。
- 生成和确认 Plan。
- 生成和确认 Tasks。
- 任务状态流转。
- 发起提测。
- 提交 Bug。
- 关闭 Bug。
- 创建发布。
- 归档基线。

AI 执行记录包括：

- 执行器。
- 任务。
- 上下文包版本。
- worktree。
- 命令日志。
- 文件变更。
- 测试结果。
- 人工确认点。
- Commit/PR/MR。

系统事件记录包括：

- 登录。
- 邀请成员。
- 分配角色。
- 配置模型。
- 配置对象存储。
- 配置 GitHub/GitLab。

V0.1 展示方式：

- 变更操作记录。

### 4.18 错误处理与失败恢复

用于保障长流程、异步任务和 AI 自动化动作失败后可见、可追溯、可恢复。

V0.1 原则：

- 失败原因必须可见。
- 失败现场尽量保留。
- 关键任务支持重试。
- 异步任务进入死信后可查看和重放。
- AI 执行失败不自动清理 worktree。
- 用户能选择继续、回滚、重试或清理。

常见失败场景：

- 文档解析失败。
- AI 生成 Proposal/Spec/Plan/Tasks 失败。
- 一致性检查失败。
- Codex/Claude Code 执行中断。
- 命令执行失败。
- 测试失败。
- Commit/Push 失败。
- PR/MR 创建失败。
- GitHub/GitLab Webhook 处理失败。
- SSE 连接断开。

AI 开发失败处理：

- 标记 AI 会话为执行失败。
- 展示失败原因。
- 保留 worktree。
- 保留终端日志。
- 保留 Diff。
- 保留测试结果。
- 支持继续执行。
- 支持重新选择执行器。
- 支持回滚或清理 worktree。

文档解析失败处理：

- 标记文件为解析失败。
- 展示失败原因。
- 支持重新解析。
- 支持切换解析器。
- 支持手动上传 Markdown/Text 替代解析结果。

队列任务失败处理：

- 自动重试固定次数。
- 超过重试次数进入死信队列。
- 支持查看失败任务。
- 支持手动重放。
- 支持记录失败上下文和错误堆栈。

SSE 断线处理：

- 支持断线重连。
- 支持 last-event-id 补偿。
- 不依赖轮询补偿主要事件。

## 5. 第一阶段 MVP 建议

第一阶段重点打通完整闭环，而不是做齐所有高级能力。

V0.1 目标只关注一条主链路：

```text
从需求/变更输入
到 SDD 生成和评审
到任务进入 Kanban
到客户端 AI 开发
到 PR/MR
到提测/Bug
到变更归档
```

建议 MVP 主流程：

```text
创建项目
-> 上传需求资料或创建变更
-> 探索/澄清需求
-> 生成 Proposal
-> 生成 Spec/Plan/Tasks
-> 一致性检查和人工评审
-> 任务进入 Kanban
-> 桌面客户端拉起 AI 开发
-> 验证/收敛遗漏任务
-> 开发完成后发起提测
-> 测试提交 Bug
-> 修复并关闭
-> 归档变更并更新基线 Spec
-> 生成发布说明和项目总结
```

V0.1 必须包含：

- 工作区/项目空间。
- 项目角色与成员。
- 项目总览 Dashboard。
- 创建变更。
- 项目资料库/知识库上传。
- Context Engineering 上下文装配。
- Spec 中心。
- SDD 工作流：Proposal -> Spec -> Plan -> Tasks。
- 轻量评审流。
- 任务 Kanban。
- 桌面客户端。
- 代码知识库轻量扫描。
- Git worktree 开发隔离。
- Codex / Claude Code 原生 CLI 编排。
- AI 执行记录。
- 权限确认模式。
- GitHub/GitLab PR/MR。
- 轻提测单。
- Bug 管理。
- 变更归档。
- 我的待办/通知。

V0.1 轻做：

- 知识库：先做上传、解析、摘要、全文检索、引用。
- 上下文工程：先做结构化关系装配、上下文包预览、引用溯源，可选语义推荐。
- 代码知识库：先做技术栈、结构、命令、关键模块识别。
- Dashboard：先做状态和风险摘要。
- 发布：先做轻量版本发布，不做复杂发布系统。
- 权限：先做项目角色 + 关键动作权限，不做复杂 RBAC。
- 通知：先做站内通知/客户端通知，第三方通知后置。

V0.1 暂不做：

- 移动 App。
- 完整组织架构。
- 复杂自定义工作流。
- 多人实时协同编辑。
- 复杂测试管理平台。
- 自动部署平台。
- 复杂数据报表。
- 企业级审计系统。
- 知识图谱。
- 自研 AI 编码 Agent。
- 完整 CI/CD 系统。
- 完整 RAG 平台。
- 自动无审核上下文注入。

V0.1 最小演示路径：

```text
创建项目
-> 添加成员角色

-> Tasks 进入 Kanban

-> 查看日志和 Diff
-> 人工确认 Commit/Push/PR
-> 发起提测
-> 测试提交 Bug

```

## 6. 暂定原则

- Web 端和桌面客户端尽量共享同一套核心功能。
- 桌面客户端是 Web 能力的增强版，不是简化版。
- 移动 App 后置，先做轻协作和通知。
- AI 不是独立聊天窗口，而是嵌入项目流程。
- 需求不能直接跳到开发任务，第一版主链路应经过 Spec Driven Development。
- 每个 AI 生成内容都要能追溯来源。
- 每个执行动作都要能回写项目记录。
- 第一版追求流程闭环，不追求大而全。
- 代码和基础能力尽量复用成熟开源方案，不重复造轮子。
- 开源组件必须经过许可证审查，避免版权、商用授权、二次分发和闭源集成风险。

## 7. 开源复用与合规原则

智流的核心竞争力不在于重写已有工具，而在于把项目流程、Spec、任务、代码、AI 执行、测试和发布串成闭环。因此，能复用成熟开源产品或组件的地方应优先复用。

优先复用方向：

- Web 前端框架和 UI 组件库。
- 桌面客户端框架。
- 富文本/Markdown 编辑器。
- Kanban/拖拽组件。
- 文档解析组件。
- PDF/DOCX/XLSX 解析能力。
- 向量检索和全文搜索能力。
- Git 操作库。
- 终端/进程执行组件。
- 代码 Diff 展示组件。
- 权限和认证基础设施。
- 后台任务队列。
- 日志和可观测性组件。

不建议复用或需要谨慎复用的方向：

- 许可证不清晰的项目。
- 强 Copyleft 且会影响智流闭源/商业分发的组件。
- 已停止维护且安全风险较高的组件。
- 需要修改大量源码才能满足需求的重型系统。
- 与智流核心流程强绑定、反而限制产品体验的完整项目管理系统。

许可证原则：

- 优先选择 MIT、Apache-2.0、BSD 等商业友好的许可证。
- 使用 GPL、AGPL、SSPL、Elastic License、BUSL 等许可证前必须单独评估。
- 前后端依赖、桌面端依赖、服务端依赖、CLI 工具和嵌入式组件要分别审查。
- 不能只看代码仓库许可证，还要看依赖树、字体、图标、模型、文档模板和素材授权。
- 所有引入的开源组件需要记录名称、用途、版本、许可证、仓库地址和替代方案。
- 如果后续支持私有化部署或商业闭源发行，必须提前确认许可证不会造成分发风险。

复用边界：

- 对通用基础能力优先复用开源。
- 对智流核心业务流程保持自有实现，例如 SDD 工作流、变更归档、AI 执行记录、任务与 Spec 关联。
- 对 Codex、Claude Code 等原生 Agent 采用编排和集成方式，不自研替代。
- 对第三方系统优先做集成，不直接复制其完整产品形态。

## 8. 技术选型原则

技术选型需要结合团队长期维护能力。当前偏好和基础能力：

- 擅长 Java / Spring。
- 擅长 Vue。
- 当前为个人开发，可以主动学习新的技术栈，但不应为了追新牺牲交付效率和长期维护性。

选型原则：

- 优先选择团队能长期维护的技术。
- 对智流核心业务后端，优先考虑 Java/Spring 生态。
- 对 Web 和桌面客户端，优先讨论 Vue 与 React 的取舍。若目标是长期产品能力和生态复用，V0.1 倾向 React + Vite + TypeScript；若目标是最快交付，可回退到 Vue 3 + Vite + TypeScript。
- 对本地 Agent 编排、终端、worktree、Git 和进程管理，优先选择生态成熟、开源许可清晰的方案。
- 技术栈可以跨语言，但边界要清晰，避免一个小团队维护过多运行时和框架。

### 8.1 Web 前端选型

V0.1 确定采用：

```text
React + Vite + TypeScript
```

选择原因：

- 智流前端更接近开发工具和复杂工作台，而不是普通后台表单。
- React 生态在编辑器、Diff、终端、拖拽、复杂布局、AI 会话流、Electron 复用方面轮子更多。
- Vite 比 Next.js 更适合第一版登录后工作台，不引入不必要的 SSR/SEO 复杂度。
- Web 和桌面客户端可以复用大量 UI、状态管理和业务组件。

建议配套：

- 路由：React Router。
- 服务端状态：TanStack Query。
- 本地状态：Zustand。
- 样式：Tailwind CSS。
- 基础组件：shadcn/ui + Radix UI。
- 图标：Lucide。
- 表单：React Hook Form + Zod。
- 表格：TanStack Table。
- 拖拽：dnd-kit。
- 终端：xterm.js。
- 代码查看/编辑：Monaco Editor。

Vue 3 + Vite + TypeScript 作为备选方案保留，但不作为 V0.1 首选。

### 8.2 桌面客户端选型

候选方案：

- Electron：使用 Chromium + Node.js，适合复杂开发工具、终端、进程管理、Git、worktree 和原生 CLI 编排。
- Tauri：使用系统 WebView + Rust，体积小、安全模型好，适合长期产品化，但需要引入 Rust 侧能力。
- Wails：使用系统 WebView + Go，适合用 Go 编写本地能力的轻量桌面应用，可作为 Electron 的轻量替代候选。

V0.1 推荐采用：

```text
Electron + React + Vite + TypeScript
```

选择原因：

- 智流桌面客户端核心是本地开发编排，不只是展示 Web 页面。
- V0.1 需要稳定处理 Codex/Claude Code CLI、终端日志、进程生命周期、Git worktree、Diff、文件系统和本地权限确认。
- Electron 的 Node 生态在 node-pty、xterm.js、进程托管、Git 调用、自动更新和开发工具类应用上更成熟。
- Web 前端已采用 React + Vite，Electron 可以最大化复用 UI 和前端工程。
- 个人开发阶段应优先降低首版集成风险。

Tauri 适合作为 V1 后的优化候选：

- 当客户端功能稳定后，如果明显受包体积、资源占用、安全隔离影响，可以评估 Tauri 重构或并行版本。
- Tauri 需要维护 Rust 侧本地能力，对个人开发者会增加学习和调试成本。

Wails 纳入候选但不作为 V0.1 首选：

- 优点：Go 编写本地能力，单体应用体验好，轻量，适合进程、文件、Git 等本地操作。
- 风险：会引入 Go 技术栈；如果后端主栈选择 Java/Spring，桌面端再引入 Go，会增加个人维护面。
- Wails v3 仍处于 Alpha 阶段，若采用需谨慎评估稳定性；Wails v2 可作为稳定候选，但复杂多窗口和开发工具类生态仍需验证。

桌面客户端选型结论：

```text
V0.1 首选 Electron
Tauri 作为长期轻量化候选
Wails 作为 Go 本地能力候选，不进入首版主路径
```

### 8.3 后端选型

V0.1 确定采用：

```text
Spring Boot 3.x + Java 21
```

架构形态：

```text
模块化单体起步，按高并发可演进架构设计
```

选择原因：

- 智流后端核心是项目、权限、SDD、评审、任务、提测、Bug、发布、通知等复杂业务流程，Spring Boot 适合长期维护。
- 当前开发者熟悉 Java/Spring，有利于快速建立稳定后端。
- 前端和桌面客户端已采用 React/Electron，后端没必要为了统一语言强行选择 Node/NestJS。
- 模块化单体比微服务更适合个人开发阶段，但代码边界和数据边界需要提前设计。

建议基础组件：

- Web/API：Spring Boot Web。
- 安全：Spring Security。
- 数据库：PostgreSQL。
- ORM/数据访问：MyBatis Plus。
- 数据库迁移：Flyway。
- 缓存/限流/分布式锁：Valkey。
- 异步任务/队列：RabbitMQ。
- API 文档：OpenAPI。
- 对象存储：S3 兼容接口。

推荐模块边界：

- auth：认证与授权。
- workspace：工作区。
- project：项目。
- member：项目成员与角色。
- change：变更管理。
- spec：Spec 中心。
- workflow：SDD 状态流与评审。
- task：任务 Kanban。
- ai-session：AI 执行记录。
- git-integration：GitHub/GitLab 集成。
- test：提测。
- bug：Bug 管理。
- release：发布与归档。
- notification：通知与待办。
- knowledge：知识库。

高并发与可扩展性原则：

- API 服务尽量无状态，支持水平扩展。
- 上传、文档解析、可选语义索引、Git 同步、通知生成等重任务异步化。
- Dashboard、Kanban、我的待办等高频读场景优先设计缓存和读模型。
- AI 执行日志采用事件流/增量写入，不把长耗时任务阻塞在 HTTP 请求里。
- GitHub/GitLab Webhook 进入队列后异步处理。
- 文件走对象存储，不直接压在应用服务器本地磁盘。
- 数据库表设计提前考虑索引、分页、归档和租户隔离。
- 先避免微服务化，但代码模块、数据库边界和事件边界要清楚。

未来可拆分服务：

- knowledge-service：知识库和检索。
- ai-session-service：AI 执行记录和日志流。
- notification-service：通知与待办。
- file-parse-service：文件解析和索引。
- git-integration-service：GitHub/GitLab 同步。

### 8.4 数据库与检索选型

V0.1 确定采用：

```text
PostgreSQL，预留 pgvector
```

选择原因：

- PostgreSQL 同时适合复杂关系数据、事务、JSONB、全文检索和扩展能力。
- 智流的项目、变更、Spec、任务、评审、Bug、发布、通知都需要可靠事务和清晰的数据建模。
- pgvector 可以作为可选语义推荐能力，承载知识片段、Spec 条目、代码知识和历史变更的向量检索。
- 一个数据底座更适合个人开发和首版部署。

主要数据类型：

- 关系数据：工作区、项目、成员、角色、变更、任务、评审、Bug、发布。
- 文档数据：Spec、Proposal、Plan、Tasks、测试用例、发布说明。
- 半结构化数据：AI 执行事件、Git webhook payload、配置快照、上下文包元数据。
- 可选向量数据：知识库片段、Spec 条目、代码知识、历史变更。

检索策略：

- 精确查询：PostgreSQL 普通索引。
- 全文检索：PostgreSQL full text search。
- 语义推荐：可选使用 pgvector。
- 上下文装配：结构化关系 + 全文检索 + 可选语义推荐 + 项目/权限/标签过滤。

扩展原则：

- V0.1 不引入独立向量库。
- V0.1 不建设完整 RAG 平台。
- V0.1 的主链路以 Context Engineering 为核心，优先使用结构化关系、源文档、全文检索和可追溯引用。
- pgvector 保留为可选推荐能力，不默认把召回片段直接注入 AI 开发上下文。
- 当知识库规模、检索延迟或索引更新压力明显上升时，再评估 Qdrant、Milvus、Weaviate 等独立向量库。
- 检索必须带项目权限过滤，避免跨项目泄露资料。
- 文档解析、片段化、可选 Embedding 生成和索引更新走异步任务。

### 8.5 数据访问层选型

V0.1 确定采用：

```text
MyBatis Plus
```

选择原因：

- 智流存在大量带权限、状态、项目、负责人、标签、时间范围的列表查询。
- Kanban、我的待办、Dashboard、变更列表、Bug 列表等场景需要较强 SQL 可控性。
- MyBatis Plus 可以兼顾基础 CRUD 效率和复杂 SQL 可维护性。
- 相比重 ORM，MyBatis Plus 更适合对查询性能、索引命中和分页行为做精细控制。

使用原则：

- 简单 CRUD 使用 MyBatis Plus。
- 复杂查询使用 XML 或显式 SQL，不把复杂业务查询硬塞进 Wrapper。
- 所有跨项目数据查询必须带权限和项目范围条件。
- 高频列表必须设计索引和分页策略。
- 重要状态变更使用事务控制。
- 数据库结构变更通过 Flyway 管理。

### 8.6 缓存与消息队列选型

V0.1 确定采用：

```text
缓存：Valkey
消息队列：RabbitMQ
```

Kafka 作为后续规模化事件流候选，不进入 V0.1 主路径。

Valkey 用途：

- 缓存。
- 限流。
- 分布式锁。
- 短期状态。
- 会话或验证码等临时数据。
- Dashboard、Kanban、我的待办等高频读场景的缓存支撑。

选择 Valkey 的原因：

- 与 Redis 生态兼容度高。
- BSD 许可证更适合商业产品和私有化部署。
- 避免 Redis 新许可证带来的合规不确定性。

RabbitMQ 用途：

- 文档解析任务。
- 可选 Embedding 生成任务。
- 知识库索引任务。
- GitHub/GitLab Webhook 异步处理。
- 通知生成。
- 发布说明生成。
- AI 执行记录异步落库或汇总。
- 失败重试和死信队列。

选择 RabbitMQ 的原因：

- 智流 V0.1 的异步场景更像任务队列，而不是海量事件流。
- RabbitMQ 支持可靠投递、路由、重试、死信和延迟处理。
- Spring Boot 集成成熟。
- MPL 2.0 许可证相对友好。

Kafka 后续适用场景：

- 海量 AI 执行事件流。
- 审计日志管道。
- 实时指标分析。
- 多消费者事件回放。
- 大规模行为分析。

V0.1 不引入 Kafka 的原因：

- 部署和运维复杂度高于当前阶段需求。
- 个人开发阶段会增加系统维护面。
- 当前核心需求是任务异步处理和可靠重试，RabbitMQ 更合适。

### 8.7 文档解析与 OCR 选型

V0.1 不接入完整 RAG 平台，采用轻量文档解析工具链，为 Context Engineering 提供源文档解析结果。

确定方案：

```text
主解析：Microsoft MarkItDown
增强解析：Docling
Java 兜底/元数据检测：Apache Tika
OCR：后续可选接入 PaddleOCR
```

MarkItDown 用途：

- 作为 V0.1 默认文档解析工具。
- 将 PDF、Word、Excel、PPT、HTML、CSV、JSON、图片等转为 Markdown。
- 输出结果用于全文搜索、摘要、引用、上下文装配和 Spec 生成。
- MIT 许可证，商用友好。

Docling 用途：

- 作为复杂文档增强解析能力。
- 用于复杂 PDF、表格、版面结构和需要更强文档理解的场景。
- 输出 Markdown/JSON/HTML 等结构化结果。
- MIT 许可证，具体模型和依赖需单独审查。

Apache Tika 用途：

- Java/Spring 侧兜底解析。
- 文件类型识别。
- 元数据抽取。
- 普通文本抽取。
- Apache-2.0 许可证。

OCR 策略：

- V0.1 默认不强制开启 OCR。
- 扫描 PDF、图片需求、截图证据等场景再进入 OCR 队列。
- OCR 结果必须标记来源和置信度。
- PaddleOCR 作为优先候选，Tesseract 作为轻量兜底候选。

推荐处理流程：

```text
上传文件
-> 保存对象存储
-> 写入文件元数据
-> RabbitMQ 投递解析任务
-> MarkItDown 解析为 Markdown
-> 必要时使用 Docling 增强解析
-> 必要时使用 Tika 兜底/抽取元数据
-> 必要时进入 OCR 队列
-> 保存解析结果和结构化片段
-> 进入全文索引和上下文装配
```

- accessKey。

### 8.9 认证与权限选型

V0.1 认证与权限采用：

```text
Sa-Token + 项目上下文权限模型
```

Sa-Token 负责：

- 登录态。
- Token 签发与校验。
- 鉴权拦截。
- 权限注解能力。
- 会话管理。
- Token 吊销/踢下线。
- 与 Valkey 集成存储登录状态。

智流负责：

- 用户。
- 工作区成员。
- 项目成员。
- 项目角色。
- 角色权限码。
- 项目上下文权限解析。
- 关键动作权限策略。

项目成员角色原则：

- 用户加入项目后默认角色为访客。
- 项目经理负责分配和调整项目角色。
- 权限基于“项目成员 -> 项目角色 -> 权限码”计算。
- 不做复杂组织架构权限继承。

推荐项目角色：

```text
访客
项目经理
产品经理
研发经理
研发人员
测试人员
运维/交付人员
```

推荐权限实现方式：

- Sa-Token 用于基础登录和认证。
- 智流自定义项目上下文权限注解，例如 `@ProjectPermission("change.spec.approve")`。
- 注解拦截器从请求中解析 `projectId`，或通过 `changeId`、`taskId`、`bugId` 反查项目。
- 拦截器统一判断当前用户在该项目内的角色是否拥有权限码。
- 业务代码只处理业务状态合法性，不散落角色权限判断。

示例：

```java
@ProjectPermission("change.plan.approve")
public void approvePlan(...) {
    // 这里只处理 Plan 是否处于可审批状态等业务规则
}
```

典型权限码：

```text
change.proposal.approve
change.spec.approve
change.plan.approve
change.tasks.approve
task.ai.execute
ai.risk.medium.approve
ai.risk.high.approve
git.pr.create
test.pass
bug.close
release.lightweight.create
release.baseline.create
spec.baseline.archive
```

### 8.10 AI 模型接入层选型

V0.1 不自研模型接入框架，优先复用成熟开源组件。

确定方案：

```text
主方案：Spring AI
备选方案：LangChain4j
后续候选：LiteLLM / 独立 AI Gateway
```

Spring AI 用途：

- Chat Model 调用。
- Streaming 输出。
- Structured Output。
- Tool/Function Calling。
- Embedding Model 接入。
- PGVector 集成。
- 观测与调用记录。
- 后续可扩展 MCP、RAG、评测等能力。

选择 Spring AI 的原因：

- 与 Spring Boot 体系天然契合。
- Apache-2.0 许可证，商用友好。
- 提供跨模型 provider 抽象，减少自研适配成本。
- 支持 OpenAI、Anthropic、DeepSeek、Ollama、Google、Azure OpenAI 等多类模型 provider。
- 支持 Embedding、PGVector、结构化输出和工具调用，适合智流的 Proposal、Spec、Plan、Tasks 生成。

LangChain4j 作为备选：

- Apache-2.0 许可证。
- Java/JVM 生态成熟。
- Provider、Tool Calling、Agent、RAG 和 Vector Store 支持丰富。
- 如果 Spring AI 在某些 provider 或 agent 能力上不满足需求，可以按模块引入 LangChain4j。

LiteLLM / AI Gateway 策略：

- 不作为 V0.1 内置核心依赖。
- 后续如果需要统一多模型路由、预算控制、fallback 和团队级模型网关，可以评估独立部署。
- 若引入，必须固定版本、使用可信镜像、做供应链安全审查。

智流自有边界：

- 不自研通用 LLM SDK。
- 不自己维护大量厂商 API 适配。
- 智流只维护业务编排层，例如任务类型、prompt 模板、上下文包、调用记录、权限和成本统计。
- 模型配置放在工作区级别，项目可以覆盖部分配置。

模型配置建议：

```text
provider
base_url
api_key
default_chat_model
default_reasoning_model
default_embedding_model
temperature
max_tokens
timeout
rate_limit
cost_limit
```

任务到模型的路由建议：

- 需求摘要、分类、发布说明：快速/低成本模型。
- Proposal、Spec、Plan、Tasks：强推理模型。
- AI 开发：优先走客户端原生 Codex/Claude Code。
- 可选语义推荐：Embedding 模型，默认不作为强事实来源。

### 8.11 工作流与状态机选型

V0.1 不引入 Flowable、Camunda、Temporal 等重型工作流引擎。

确定方案：

```text
自研受限状态机
```

边界：

- 不做通用工作流平台。
- 不做 BPMN。
- 不做可视化流程编排。
- 不做任意条件网关/并行网关。
- 只管理智流自己的固定业务流程。

适用对象：

- 变更状态。
- Proposal/Spec/Plan/Tasks 评审状态。
- 任务 Kanban 状态。
- 提测状态。
- Bug 状态。
- 发布状态。
- AI 执行状态。

状态流转原则：

- 每个状态变更必须由明确业务动作触发。
- 不允许随意传状态字符串跳转。
- 每个动作必须检查当前状态、权限、前置条件。
- 每次流转必须记录 transition log。
- 流转成功后发布 domain event。
- 后置动作通过事件异步触发。

建议核心组件：

```text
StateTransitionService
TransitionDefinition
TransitionGuard
TransitionAction
TransitionLog
DomainEvent
```

示例业务动作：

```text
change.submitProposal
change.approveSpec
change.approvePlan
change.confirmTasks
task.startDevelopment
task.requestTest
task.completeTest
bug.reopen
release.archiveBaseline
```

流转步骤：

```text
1. 校验当前状态是否合法
2. 校验用户是否有权限
3. 校验前置条件是否满足
4. 执行业务状态变更
5. 写入 transition_log
6. 发布 domain_event
7. 异步触发后置动作
```

有限配置：

- 是否启用待发布状态。
- 是否允许跳过测试中。
- 是否进入待提测必须绑定 PR/MR。
- 是否进入开发前必须完成全部评审。

后续只有当流程复杂到需要用户自定义编排、复杂审批流或跨系统长事务时，再评估引入专业工作流引擎。

### 8.12 实时通信与事件推送选型

V0.1 不使用 HTTP 轮询作为主要实时方案，也不使用数据库轮询作为主要事件方案。

确定方案：

```text
服务端内部事件：RabbitMQ
服务端到 Web/桌面客户端：SSE
桌面客户端本地进程到 UI：Electron IPC
```

适用场景：

- AI 执行日志。
- 命令执行状态。
- 任务状态变化。
- 评审状态变化。
- PR/MR 状态变化。
- 待办通知。
- Dashboard 局部更新。
- Bug 指派和验证结果。

服务端事件流程：

```text
业务动作
-> 写数据库
-> 发布 domain_event
-> 投递 RabbitMQ
-> 实时推送服务消费事件
-> 通过 SSE 推给订阅用户
```

AI 执行日志流程：

```text
Electron Main 托管 Codex/Claude Code
-> 捕获 stdout/stderr/命令事件
-> Electron IPC 推给客户端 UI
-> 异步上报后端
-> 后端通过 SSE 推给其他观察者
```

SSE 使用原则：

- 用于服务端单向推送。
- 每个用户建立自己的事件连接。
- 事件必须带工作区/项目/资源权限过滤。
- 支持断线重连和 last-event-id。
- 不承载大文件或超大日志全文，只推增量事件和状态变化。

暂不使用 WebSocket 的原因：

- V0.1 不做多人实时协同编辑。
- 当前主要需求是服务端到客户端的事件推送。
- SSE 更简单，和 HTTP 基础设施兼容性更好。

后续如果出现多人实时编辑、多人 Kanban 协作、双向低延迟控制 AI 会话等需求，再评估 WebSocket。

## 9. 产品体验与界面风格原则

智流不应做成传统企业后台管理系统，也不应呈现过重的 SaaS 管理台风格。

目标体验：

- 像研发项目组的日常工作台。
- 像 Spec、任务、代码和 AI 执行的项目操作系统。
- 专业、克制、高效，但不能呆板。
- 信息密度适中，服务高频工作流。
- 优先让用户知道“下一步该做什么”。

应避免：

- 大面积营销式首页。
- 传统企业后台的菜单堆叠感。
- 过重的表格管理系统气质。
- 过多卡片套卡片。
- 花哨但无信息价值的大屏。
- 单一蓝紫渐变或模板化 SaaS 视觉。
- 把所有功能做成“管理配置页”。

推荐界面方向：

- 项目为主的工作台布局。
- 左侧项目/模块导航，中间主工作区，右侧上下文/AI/待办面板。
- Spec、Plan、Tasks、Diff、日志等内容优先采用编辑器/工作区式布局。
- Dashboard 更像项目作战台，而不是管理报表大屏。
- Kanban 展示执行状态，同时能按变更、负责人、风险和 AI 状态筛选。
- AI 执行界面应像可审计的开发会话，而不是普通聊天窗口。
- 客户端体验应接近开发工具，Web 体验应接近协作工作台。

视觉气质关键词：

```text
研发工作台
项目操作系统
沉稳但不老气
信息清晰
流程感强
上下文丰富
行动导向
```
